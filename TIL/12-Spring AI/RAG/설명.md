# RAG 이해하기

> Document 개념 → [Document.md](../Document.md)

```java
private List<Document> getDocuments(String query) {
    return vectorStore.similaritySearch(
        SearchRequest.builder()
            .query(query)
            .similarityThreshold(0.7) // 유사도 임계치
            .topK(50) // 유사도가 높은 상위 상품 우선적으로 추출
            .build()
    );
}
```
vectorStore.similaritySearch(): 실제로 DB에서 '거리 계산(유사도 계산)'을 수행하여
가장 가까운 `Document` 객체들을 리스트 형태로 반환한다.

## 내부 프로세스 (RAG의 핵심)
1. **Embedding**: `EmbeddingModel`이 query 문자열을 벡터(숫자 배열)로 변환
2. **Similarity Calculation**: DB 내의 모든 벡터와 `query` 벡터 사이의 거리를 계산
3. **Filtering**: 거리가 가장 가까운 상위 N개를 뽑기
4. **Mapping**: 검색된 벡터에 매칭되는 원문 텍스트와 메타데이터를 `Document` 객체에 담아 반환

## topK vs similarityThreshold

두 옵션 모두 결과 수를 줄이지만 동작 방식이 다르다.

| 옵션 | 동작 | 특징 |
|---|---|---|
| `topK(N)` | 유사도 순으로 무조건 상위 N개 반환 | 결과 수가 항상 고정 |
| `similarityThreshold(0.7)` | 유사도가 기준값 이상인 것만 반환 | 결과 수가 가변적 |

같이 쓰면 **AND 조건**으로 동작한다.
→ "유사도 0.7 이상인 것 중에서 상위 50개"

```java
SearchRequest.builder()
    .query(query)
    .similarityThreshold(0.7) // 기준 미달은 아예 제외
    .topK(50)                 // 그 중 상위 50개
    .build()
```

## 주의사항
- `topK`가 너무 크면 LLM의 Context Window(토큰 제한)를 초과하거나 노이즈가 섞여 답변 품질이 떨어진다. 보통 **3~10개** 사이를 권장.
- 데이터를 저장할 때 사용한 `EmbeddingModel`과 검색 시 사용하는 `EmbeddingModel`이 다르면 벡터 공간이 달라져 검색 정확도가 크게 저하된다. 반드시 동일한 모델을 사용해야 한다.
