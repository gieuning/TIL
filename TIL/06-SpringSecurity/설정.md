문제점 3가지

1. 비밀번호 비교 안 됨 - 주석만 있고 passwordEncoder.matches() 없음
2. authenticationManager, securityContextRepository 주입은 했는데 사용 안 함 - 현재 수동으로 DB 조회만 하는 방식
3. HttpServletRequest, HttpServletResponse 파라미터 없음 - 세션에 저장하려면 필요

  ---
다음 순서

AuthenticationManager를 제대로 쓰려면 Spring Security가 내부적으로 loadUserByUsername(email)을 호출할 때 처리할 클래스가
필요해요. 그게 지금 없어요.

① CustomUserDetails 만들기 (UserDetails 구현체 - userId, email 등 담는 그릇)

② CustomUserDetailsService 만들기 (UserDetailsService 구현체 - DB에서 유저 찾아서 CustomUserDetails 반환)

이 둘이 있어야 authenticationManager.authenticate() 가 동작하고, 그 다음에

③ login 메서드 완성
- authenticationManager.authenticate() 호출
- SecurityContext 만들어서 securityContextRepository.saveContext()로 세션 저장
- principal에서 CustomUserDetails 꺼내서 LoginResponse 반환
- 메서드 파라미터에 HttpServletRequest, HttpServletResponse 추가

SecurityContextRepository

역할: "인증 정보를 어디에 저장/불러올지" 담당

현재 SecurityConfig에서 HttpSessionSecurityContextRepository로 설정했는데, 이는 세션에 저장한다는 뜻입니다.

로그인 시 → securityContextRepository.saveContext(context, request, response)
→ 세션에 인증 정보 저장

다음 요청 시 → 자동으로 세션에서 꺼내서 SecurityContext 복원
→ controller에서 Authentication 바로 꺼낼 수 있음

즉, 로그인 한 번 하면 이후 요청에서 "이 사람 로그인 한 사람이다"라는 걸 세션으로 유지해주는 역할입니다.

CustomUserDetails / CustomUserDetailsService

왜 필요한가?

Spring Security의 기본 UserDetails는 username, password, authorities만 담을 수 있어요. 근데 로그인 후 userId도 꺼내 써야
합니다 (장바구니 등에서 userId 필요).

기본 UserDetails → username, password, authorities만 있음
CustomUserDetails → userId, email 등 추가 필드 담을 수 있음

**CustomUserDetailsService**는 AuthenticationManager가 내부적으로 호출하는 서비스예요:

UserDetailsService (인터페이스)
loadUserByUsername(String email)
→ UserRepository.findByEmail(email)
→ User 엔티티를 CustomUserDetails로 변환해서 반환

  ---
전체 흐름 정리

POST /api/auth/login

1. AuthService.login() 호출
2. authenticationManager.authenticate(email, password)
   ↓ 내부적으로
3. CustomUserDetailsService.loadUserByUsername(email)
   → DB에서 유저 조회 → CustomUserDetails 반환
   ↓
4. PasswordEncoder로 비밀번호 비교
   → 성공: Authentication 객체 반환
   ↓
5. SecurityContext에 Authentication 저장
6. securityContextRepository.saveContext() → 세션에 저장
   ↓
7. Authentication에서 CustomUserDetails 꺼냄
8. userId, email → LoginResponse 반환

이 흐름 이해되면 CustomUserDetails → CustomUserDetailsService 순서로 만들면 됩니다.



여기서 쓰는 Redis는 세션 저장소로 쓰는 거라 캐시 패턴이랑은 다른 용도예요.

  ---
캐시 패턴들은 언제 쓰나

DB 조회 결과를 Redis에 캐싱할 때 사용해요.

Write-Through - 캐시랑 DB 동시에 씀
저장 요청 → Redis 저장 + DB 저장 (동시)

Write-Back (Write-Behind) - 캐시에 먼저 쓰고 나중에 DB에 씀
저장 요청 → Redis 저장 → (비동기) → DB 저장
DB 부하 줄일 수 있지만 Redis 죽으면 데이터 유실 위험

Write-Around - DB에만 쓰고 캐시는 스킵
저장 요청 → DB 저장 (캐시 건너뜀)
읽기 요청 → 캐시 미스 → DB 조회 → 캐시 저장
자주 안 읽히는 데이터에 적합

Cache-Aside (Lazy Loading) - 읽을 때 캐시 없으면 DB에서 가져와서 캐시에 저장
읽기 요청 → 캐시 확인 → 없으면 DB 조회 → 캐시 저장 → 반환

  ---
현재 프로젝트에서는

지금은 Redis를 세션 저장소로만 써서 이 패턴들이 해당 없어요.

이 패턴들은 나중에 예를 들어 상품 정보나 사용자 정보를 자주 조회할 때 DB 부하 줄이려고 Redis에 캐싱하는 상황에서 쓰게
됩니다.

Cache-Aside 예시 (가장 많이 씀)

상품 정보 조회할 때:

@Service
@RequiredArgsConstructor
public class ProductService {

      private final ProductRepository productRepository;
      private final RedisTemplate<String, Object> redisTemplate;

      public ProductResponse getProduct(Long productId) {
          String cacheKey = "product:" + productId;

          // 1. 캐시 먼저 확인
          ProductResponse cached = (ProductResponse) redisTemplate.opsForValue().get(cacheKey);
          if (cached != null) {
              return cached; // 캐시 히트 → DB 안 감
          }

          // 2. 캐시 미스 → DB 조회
          Product product = productRepository.findById(productId)
              .orElseThrow(() -> new DomainException(DomainExceptionCode.NOT_FOUND_PRODUCT));

          ProductResponse response = ProductResponse.from(product);

          // 3. 캐시에 저장 (1시간 후 만료)
          redisTemplate.opsForValue().set(cacheKey, response, 1, TimeUnit.HOURS);

          return response;
      }
}

첫 번째 요청  → 캐시 미스 → DB 조회 → 캐시 저장 → 반환
두 번째 요청~ → 캐시 히트 → DB 안 감 → 바로 반환 (빠름)

언제 쓰면 좋은가

자주 조회되고 (읽기 많음)
자주 안 바뀌는 데이터 (상품 정보, 카테고리 등)

반대로 자주 바뀌는 데이터는 캐시랑 DB 사이 불일치 문제가 생길 수 있어서 주의해야 해요.




기존 방식 (메모리 세션) 문제점

서버 1 (로그인한 세션 있음)
서버 2 (세션 없음)
서버 3 (세션 없음)

사용자가 서버 1에 로그인 → 서버 2로 요청 → "로그인 안됨"

서버가 여러 대일 때 세션이 공유가 안 돼요. 그리고 서버 재시작하면 세션 다 날아가요.

  ---
Redis 세션 저장소 장점

① 서버 여러 대여도 세션 공유
서버 1, 2, 3 → 전부 Redis에서 세션 꺼냄
어느 서버로 요청이 가도 로그인 유지

② 서버 재시작해도 세션 유지
서버 재시작 → Redis는 살아있음 → 세션 그대로

③ 세션 만료 관리가 편함
Redis TTL로 자동 만료 처리

N+1 사전에 발생 예상하는 기준

"목록 조회 + 연관관계 접근" 조합이 나오면 바로 의심해요.

1번 쿼리  → 목록 전체 조회 (SELECT * FROM product)
N번 쿼리  → 각 product마다 category 조회
(SELECT * FROM category WHERE id = 1)
(SELECT * FROM category WHERE id = 2)
(SELECT * FROM category WHERE id = 3) ...

체크리스트

✅ findAll() 또는 목록 조회가 있는가?
✅ 조회한 목록을 루프 돌리는가?
✅ 루프 안에서 연관 객체에 접근하는가? (getCategory(), getUser() 등)

세 개 다 해당하면 N+1 발생해요.
