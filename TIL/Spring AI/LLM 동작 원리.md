# LLM 동작 원리

## 전체 흐름

```
입력 → 토큰화 → 임베딩 → 트랜스포머 처리 → 다음 토큰 예측 → 반복 → 응답 출력
```

## 토큰화

문장을 모델이 이해할 수 있는 최소 단위로 분리. 단어 단위가 아니라 **자주 등장하는 패턴** 기준으로 쪼갬 (BPE 방식).

```
"Spring Boot란?" → ["Spring", "Boot", "란", "?"]
"unhappiness"   → ["un", "happiness"]   ← 자주 쓰이는 조각으로 분리
```

각 토큰은 고유한 정수 ID로 매핑됨

```
"Spring" → 15043
"Boot"   → 20521
"란"     → 30003
"?"      → 30
```

- 영어 4글자 ≈ 1토큰, 한글 1.5~2글자 ≈ 1토큰
- 토큰 수 = 비용 → 한글이 영어보다 1.5~2배 더 소모

## 임베딩

각 토큰 ID를 고차원 숫자 벡터로 변환. 모델이 학습하면서 **의미가 유사한 단어는 가까운 벡터**를 갖도록 조정됨.

```
"king"  → [0.2, 0.8, 0.5, ...]   (수백~수천 차원)
"queen" → [0.2, 0.7, 0.5, ...]   ← 가까움
"apple" → [0.9, 0.1, 0.3, ...]   ← 멀음
```

벡터 간 거리는 **코사인 유사도**로 측정. 유명한 예시:

```
king - man + woman ≈ queen   (벡터 연산이 의미를 반영)
```

> 토큰화가 ID를 만들고, 임베딩이 그 ID를 의미 있는 숫자로 바꿔준다.
> 이후 트랜스포머는 이 벡터들을 입력으로 받아 문맥을 파악함.

## 트랜스포머 처리

| 단계 | 역할 |
|------|------|
| Self-Attention | 단어 간 관계 파악 (문맥 이해) |
| Feed Forward | 학습된 지식 인출 및 정제 |

## 다음 토큰 예측

확률이 높은 토큰을 하나씩 선택하며 문장 완성. 이 과정을 응답이 끝날 때까지 반복.

## 토큰 비용 줄이는 법

```
1. 불필요한 서술어 제거
   Bad:  "스프링 부트가 정확히 무엇인지 자세히 설명해주세요"
   Good: "Spring Boot 정의 및 특징 요약"

2. 영어 혼용
   Bad:  "자바 기반의 웹 프레임워크"
   Good: "Java Web Framework"

3. 출력 형식 지정
   "핵심 내용만 3문장 이내로 요약해줘"

4. System Message에 반복 지침 한 번만 정의
```
