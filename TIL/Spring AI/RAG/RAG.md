# RAG (Retrieval-Augmented Generation, 검색 증강 생성)

RAG는 외부의 신뢰할 수 있는 데이터 소스에서 관련 정보를 검색(Retrieval)한 뒤,  
그 내용을 바탕으로 LLM이 답변을 생성(Generation)하도록 하는 기술이다.

즉,  
**사용자의 질문(Query) + 검색된 문서(Document)**를 함께 LLM에 전달하여  
더 정확하고 최신의 답변을 생성할 수 있게 한다.

---

## RAG의 동작 방식

1. 사용자가 질문(Query)을 입력한다.
2. 질문과 관련된 문서를 Vector DB에서 검색한다.
3. 검색된 문서를 LLM에 함께 전달한다.
4. LLM이 문서를 기반으로 답변을 생성한다.

---

##  핵심 구조

Query + Retrieved Documents → LLM → Answer

---

## 중요한 포인트

- Document는 그대로 사용하지 않는다.
- 문서 크기가 크기 때문에 **청크(Chunk) 단위로 분할**하여 저장하고 검색한다.
- 질문과 관련된 **일부 청크만 필터링**하여 LLM에 전달한다.

---

## 상세 흐름

1. Document를 잘게 쪼갠다 (Chunking)
2. 각 청크를 임베딩하여 Vector DB에 저장
3. 사용자 Query도 임베딩
4. 유사도 기반으로 관련 청크 검색
5. 검색된 청크 + Query를 LLM에 전달
6. 최종 답변 생성

---

## 왜 RAG가 필요한가? (LLM의 한계)

LLM은 강력하지만 다음과 같은 한계가 있다.

### 1. 지식의 단절
- 학습 시점 이후의 최신 정보를 알 수 없음

### 2. 폐쇄적 데이터 접근 불가
- 기업 내부 문서, 개인 데이터 등은 알 수 없음

### 3. 환각(Hallucination)
- 모르는 내용을 그럴듯하게 지어낼 위험 존재

---

## 💡 예시

### ❌ RAG 미적용

사용자: "우리 회사의 2024년 휴가 정책은?"

LLM:  
"죄송하지만 해당 정보는 알 수 없습니다."

---

### ✅ RAG 적용

1. Vector DB에서 "휴가 정책" 관련 문서 검색
2. 해당 문서를 LLM에 함께 전달

LLM:  
"2024년 휴가 정책에 따르면 연차는 15일이며..."

---

## RAG의 핵심 가치

- 최신 정보 반영 가능
- 내부 데이터 활용 가능
- 환각 감소 → 신뢰도 향상

---

## CAG와의 차이 (중요)

- **RAG**: 검색(Retrieval) 기반 → 필요한 문서를 찾아서 사용
- **CAG**: 단순히 Context를 함께 넣는 방식 (검색 없음)

👉  
Document를 그냥 붙이는 것은 RAG가 아니라  
**CAG(Context Augmented Generation)에 가까움**
