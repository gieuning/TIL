# Document

데이터를 다루는 가장 기본적인 단위
단순히 문자열이 아니라, AI가 문맥을 더 잘 파악하거나 개발자가 데이터를 관리하기 쉽도록
'본문 내용'과 '부가 정보'를 하나로 묶어놓은 가방과 같다.

## 구성 요소

### 1. text (본문 내용)
- 원문 텍스트
- Vector DB에 저장될 때 이 텍스트를 기반으로 임베딩(숫자화)이 생성
- 검색 결과로 반환되었을 때 LLM에게 전달되어 답변의 근거로 사용되는 부분

### 2. metadata
- 본문 외의 해당 데이터와 관련된 추가 정보들을 `Map<String, Object>` 형태로 담는다.
- 사용 예시:
  - 이 문장이 포함된 파일의 이름 (filename)
  - 문서의 몇 페이지였는지 (page_number)
  - 작성자, 생성일, 카테고리 등
- 필터링 역할: 검색할 때 "특정 파일 내에서만 찾아줘" 혹은 "특정 날짜 이후의 데이터만 검색해줘"와 같은 조건을 걸 때 사용된다.

## 왜 Document를 쓰나?

1. **출처 추적 가능**: 검색 결과로 나온 텍스트가 "어느 문서에서 왔는지" 사용자에게 알려주기 위한 메타데이터가 필요
2. **정교한 검색**: 단순히 유사한 것만 찾는 게 아니라, "특정 프로젝트에 속한 데이터 중에서만" 유사한 것을 찾도록 필터링을 할 수 있다.
3. **유연성**: 다양한 소스에서 읽어온 데이터를 일관된 포맷으로 변환하여 Vector DB에 넣기 위함이다.

정리하자면 `Document`는 Vector DB가 저장하고 반환하는 '정보의 최소 단위'이며, 검색된 상위 N개의 결과물들이 각각 이 형태를 띠고 리스트에 담긴다.
