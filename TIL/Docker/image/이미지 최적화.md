# Docker 이미지 최적화

## .dockerignore

`docker build` 실행 시 현재 디렉터리 전체가 Docker Daemon에 빌드 컨텍스트로 전달된다.  
불필요한 파일이 많으면 빌드가 느려지고, 민감한 파일이 이미지에 포함될 수 있다.

```
.git
.env
node_modules
build
*.log
```

| 효과 | 설명 |
|---|---|
| 빌드 속도 향상 | 전달되는 컨텍스트 크기 감소 |
| 보안 | `.env`, 인증 정보 등이 이미지에 포함되는 것 방지 |
| 이미지 크기 감소 | 불필요한 파일 제외 |

> Java는 jar만 이미지에 넣는 경우가 많아 체감이 작지만,  
> Node.js / Python처럼 `node_modules`, 캐시 파일이 큰 프로젝트에서는 필수다.

## Dockerfile 배치 전략

Layer 캐싱을 최대한 활용하려면 변경 빈도에 따라 순서를 정해야 한다.

```
변경 적음 (위에 배치)
 ↓ 베이스 이미지
 ↓ 패키지/의존성 설치
 ↓ 설정 파일
변경 잦음 (아래 배치)
 ↓ 소스 코드 복사
 ↓ 실행 명령
```

### 잘못된 예 — 캐시가 매번 깨짐

```dockerfile
FROM python:3.9
WORKDIR /app
COPY .. .                          # 코드 변경 시 여기서 캐시 깨짐
RUN pip install -r requirements.txt  # 매번 재실행됨
CMD ["python", "app.py"]
```

### 올바른 예 — 의존성 캐시 유지

```dockerfile
FROM python:3.9
WORKDIR /app
COPY requirements.txt .           # 의존성 파일만 먼저 복사
RUN pip install -r requirements.txt  # 캐시 유지 (requirements.txt 변경 시에만 재실행)
COPY . .                          # 소스 코드는 나중에
CMD ["python", "app.py"]
```

## Multi-stage Build

빌드 단계와 실행 단계를 분리해서 실행 이미지에는 필요한 것만 포함시킨다.

```
빌드 이미지 (무거움)        실행 이미지 (가벼움)
소스 코드 전체         →    jar 파일만
Gradle / Maven        →    JRE
테스트 도구           →    실행 명령
```

```dockerfile
# 1단계: 빌드
FROM gradle:8-jdk21 AS builder
WORKDIR /app
COPY . .
RUN gradle build --no-daemon

# 2단계: 실행
FROM amazoncorretto:21-alpine-jdk
WORKDIR /app
COPY --from=builder /app/build/libs/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

최종 이미지에는 2단계만 포함된다. 소스 코드, Gradle, 테스트 파일은 들어가지 않는다.

## 이미지 정리

CI/CD 환경에서는 빌드마다 이미지가 쌓인다. EC2처럼 디스크가 작은 환경에서는 주기적으로 정리해야 한다.

```bash
docker images                # 현재 이미지 목록 확인
docker image prune           # 태그 없는 이미지(댕글링) 삭제
docker system prune          # 이미지 + 중지된 컨테이너 + 네트워크 정리
docker system prune -a       # 사용 중이지 않은 이미지 전체 삭제
```

> `system prune -a`는 실행 중인 컨테이너가 사용하는 이미지는 삭제하지 않지만,  
> 운영 서버에서는 실수 방지를 위해 조심해서 사용한다.
