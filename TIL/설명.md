# 엔티티 vs 서비스 책임 분리

## 엔티티에 두는 것
- 자기 상태를 바꾸는 메서드
- 자기 필드끼리의 계산
- 도메인 규칙 (예: 재고 0 이하면 주문 불가, 가격 변경 시 상태 변경)

## 서비스에 두는 것
- 여러 엔티티를 엮는 로직
- 외부 API 호출
- 트랜잭션 처리
- 저장/조회 흐름 제어

# 스프링 AOP와 Self-Invocation 문제

## 프록시 기반 동작 원리

`@Transactional`, `@Retryable`, `@Async` 같은 어노테이션은 메서드를 직접 바꾸는 게 아니라, **프록시가 바깥을 감싸서** 동작한다.

```
외부 호출 → [프록시] → 실제 메서드  ✅ @Transactional 적용됨
this.someMethod() → 실제 메서드    ❌ 프록시를 안 탐
```

## Self-Invocation 문제

`this.someMethod()`처럼 **자기 자신의 메서드를 내부에서 호출**하면 프록시를 거치지 않는다.  
→ `@Transactional`, `@Retryable`이 붙어 있어도 **효과 없음**

**해결책:** 해당 메서드를 별도 Bean으로 분리해서 외부 호출로 만든다.

> "자기 자신을 호출하지 말고 별도 빈으로 빼라"는 말이 나오는 이유가 바로 이것

### 문제 상황

```java
@Service
public class OrderService {

    public void placeOrder(Long itemId) {
        // this로 내부 호출 → 프록시를 안 탐 → @Transactional 무시됨
        this.saveOrder(itemId);
    }

    @Transactional
    public void saveOrder(Long itemId) {
        // DB 저장 로직
    }
}
```

### 해결: 별도 Bean으로 분리

```java
// 트랜잭션 처리 전담 Bean 분리
@Component
public class OrderWriter {

    @Transactional
    public void saveOrder(Long itemId) {
        // DB 저장 로직
    }
}

@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderWriter orderWriter;

    public void placeOrder(Long itemId) {
        // 외부 Bean 호출 → 프록시를 탐 → @Transactional 정상 적용
        orderWriter.saveOrder(itemId);
    }
}
```

# 외부 API 호출과 트랜잭션

외부 API 호출을 DB 트랜잭션 안에 오래 물고 있으면 **락/지연 문제**가 생길 수 있다.  
(네트워크 대기 시간 동안 트랜잭션이 열린 채로 유지됨)

**권장 패턴:**
| 작업 | 처리 방식 |
|---|---|
| 외부 API 호출 | 트랜잭션 밖 또는 별도 책임 |
| DB 저장/수정 | 필요한 범위만 짧게 `@Transactional` |

---

## 정리

> "외부 호출을 무조건 분리한다"기보다,  
> **자기 자신 호출은 프록시를 안 타서 트랜잭션/재시도가 안 먹을 수 있으니,  
> 그런 메서드는 별도 Bean으로 분리하는 경우가 많다**가 정확한 표현.
