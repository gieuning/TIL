# 동시성 (Concurrency)

콘솔창 하나 -> 하나의 세션

**동시성**: 여러 개의 작업(프로세스, 트랜잭션)이 동시에 실행되는 것

동시성은 시스템 성능에 필수적이지만, 아무런 통제 장치 없이 방치하면 데이터의 신뢰도를 파괴하는
심각한 문제를 일으킨다. 이를 **데이터 정합성이 깨졌다**고 표현한다.

> 동시에 접근하는 것 자체가 문제가 아니라, 동시 접근으로 인해 ACID(원자성, 일관성, 격리성, 지속성)가 깨지는 것이 문제다.

---

## 격리 수준(Isolation Level)

격리 수준 = **"언제 어떤 락을 얼마나 오래 잡을지"에 대한 정책**

격리 수준이 높아질수록 더 많은 락을 더 오래 잡아 **동시성은 낮아지고, 데이터 정합성은 높아진다**.

### 격리 수준별 발생 가능한 문제

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| Read Uncommitted (레벨 0) | 발생 | 발생 | 발생 |
| Read Committed (레벨 1) | 방지 | 발생 | 발생 |
| Repeatable Read (레벨 2) | 방지 | 방지 | 발생 (MySQL은 MVCC로 대부분 방지) |
| Serializable (레벨 3) | 방지 | 방지 | 방지 |

---

## 동시성 이상 현상 (Concurrency Anomalies)
### 1. Dirty Read

- **발생**: Read Uncommitted
- 커밋되지 않은 데이터를 다른 트랜잭션이 읽는 문제
- A가 롤백하면 B는 존재하지 않는 데이터를 읽은 셈이 됨

```
트랜잭션 A: 나이 25세로 수정 (미커밋)
트랜잭션 B: 나이 조회 → 25세 읽음  ← Dirty Read
트랜잭션 A: 롤백 → 실제 나이는 여전히 24세
```

---

### 2. Non-Repeatable Read

- **발생**: Read Committed
- 같은 트랜잭션 내에서 같은 행을 두 번 읽었을 때 결과가 다른 문제
- **UPDATE/DELETE**로 인한 기존 행의 변경이 원인
- 격리성 관점에서 트랜잭션은 독립적으로 실행되어야 하므로, 커밋된 데이터조차 트랜잭션 내에서는 변경되어 보이면 안 됨

```
트랜잭션 A: ID=100 사용자 나이 조회 → 25세
트랜잭션 B: ID=100 사용자 나이를 26세로 수정 후 커밋
트랜잭션 A: ID=100 사용자 나이 재조회 → 26세  ← Non-Repeatable Read
```

---

### 3. Phantom Read

- **발생**: Repeatable Read
- 같은 트랜잭션 내에서 같은 조건으로 조회했는데 행의 **개수**가 달라지는 문제
- **INSERT**로 인한 새로운 행 추가가 원인
- Repeatable Read는 기존 행의 변경은 보호하지만, 새로 삽입된 행은 막지 못함

Non-Repeatable Read와 헷갈리기 쉽지만 **무엇이 바뀌냐**가 다르다.

| | Non-Repeatable Read | Phantom Read |
|---|---|---|
| 원인 | 기존 행이 **UPDATE/DELETE** 됨 | 없던 행이 **INSERT** 됨 |
| 증상 | 같은 행의 **값**이 달라짐 | 같은 조건인데 **행 개수**가 달라짐 |

```
트랜잭션 A: age 20~30인 사용자 수 조회 → 5명

  (트랜잭션 B가 age=25인 사용자를 INSERT하고 커밋)

트랜잭션 A: age 20~30인 사용자 수 재조회 → 6명  ← Phantom Read
```

A 입장에서는 아무것도 안 했는데 **유령(Phantom)처럼 행이 하나 생겨버린** 것.

#### Repeatable Read에서 왜 막히지 않나?

Repeatable Read는 **이미 읽은 행**에만 락을 건다.

```
처음 조회 → age=25인 행(ID=1), age=28인 행(ID=2) 에 락
→ 이 두 행은 수정/삭제 불가 (Non-Repeatable Read 방지)

BUT, age=27인 완전히 새로운 행(ID=3)은 락 대상이 아님
→ INSERT는 막을 수 없음 → Phantom Read 발생
```

#### Serializable의 갭 락(Gap Lock)이 해결하는 방법

```
처음 조회 → age 20~30 범위 자체에 갭 락
→ 이 범위 안에는 아무도 INSERT 불가
→ Phantom Read 방지
```

> 핵심: **"이미 있는 행을 잠그는 것"** vs **"없는 공간(gap)을 잠그는 것"** 의 차이

## 격리 수준 상세

### Read Uncommitted (레벨 0)

- 격리 수준이 사실상 없음
- 가장 낮은 격리 수준. 커밋되지 않은 데이터도 읽는 것을 허용
- 장점: 트랜잭션 간 잠금이 거의 없어 가장 높은 성능
- 단점: 데이터 정합성에 심각한 문제 → 실무에서 거의 사용하지 않음

---

### Read Committed (레벨 1)

- **커밋된 데이터만** 읽도록 허용 → Dirty Read 해결
- 동작 방식: 읽기 시 공유 잠금(Shared Lock)을 걸지만, **읽기가 끝나면 바로 잠금 해제**
- Non-Repeatable Read, Phantom Read는 여전히 발생
- Oracle, PostgreSQL, SQL Server 등의 **기본 격리 수준**

---

### Repeatable Read (레벨 2)

- MySQL InnoDB의 **기본 격리 수준**
- 트랜잭션 시작 시점의 **스냅샷(MVCC)** 을 읽어 트랜잭션 내 데이터 일관성 유지
- 원본 데이터가 바뀌더라도 트랜잭션 내에서는 처음 읽은 값이 유지됨
- MySQL은 MVCC 덕분에 일반 SELECT에서 Phantom Read도 대부분 방지
  - 단, `SELECT ... FOR UPDATE`(Locking Read)는 최신 데이터를 읽어 Phantom Read 발생 가능

#### MVCC (Multi-Version Concurrency Control)

Repeatable Read는 이론상 Phantom Read가 발생할 수 있지만, 현대적인 DB는 **MVCC**로 이를 해결한다.

**동작 방식:**
- 트랜잭션이 데이터를 수정할 때, InnoDB는 기존 데이터를 즉시 삭제/덮어쓰지 않는다
- 변경 전 원본 데이터를 **Undo Log** 영역에 복사해두고, 각 레코드에 변경한 트랜잭션 번호를 기록
- 읽기 시점에 자신의 트랜잭션 번호 기준으로 **스냅샷(원본)을 바라본다**

> MVCC = 원본은 놔두고, 스냅샷을 바라본다

**왜 MVCC가 필요한가?**
- Repeatable Read에서 락만으로 구현하면 UPDATE할 때마다 블락이 걸림
- 블락이 많아지면 속도가 느려짐
- MVCC는 락 없이도 높은 격리성을 제공 → 성능과 격리성을 동시에 만족
- MVCC가 있는 DB(MySQL InnoDB 등)는 Phantom Read까지 해결


> Phantom Read를 완전히 방지하는 방법은 Serializable이다. MVCC는 일반 SELECT에 한해 대부분 방지하지만, `SELECT ... FOR UPDATE` 같은 Locking Read에서는 여전히 발생할 수 있다.

---

### Serializable (레벨 3)

- 가장 높은 격리 수준. 트랜잭션을 순차적으로 실행하는 것과 동일한 결과를 보장
- **갭 락(Gap Lock)** 을 사용하여 Phantom Read까지 방지
- 인덱스 레코드 사이의 **간격(Gap)** 을 잠궈 새로운 행 삽입 자체를 차단

```sql
-- age 25~30 범위 조회 시 갭 락 획득
SELECT * FROM users WHERE age BETWEEN 25 AND 30 FOR UPDATE;

-- 다른 트랜잭션에서 이 범위에 INSERT하면 대기(블로킹)
INSERT INTO users (age) VALUES (27); -- 블로킹!
```
