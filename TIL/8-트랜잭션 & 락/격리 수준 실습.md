# 트랜잭션 격리 수준 실습

## 잠금(Lock)의 종류

데이터베이스는 동시성을 제어하기 위해 다양한 종류의 잠금을 사용한다.

### MySQL 레벨 락 (스토리지 엔진 독립)

| 잠금 종류 | 설명 |
|---|---|
| 레코드 잠금 | 특정 하나의 행에만 거는 잠금. UPDATE/DELETE 시 사용. Non-Repeatable Read 방지 |
| 갭 잠금 (Gap Lock) | 데이터 행과 행 사이의 '간격'에 거는 잠금. 이 간격에 새로운 데이터가 INSERT 되는 것을 방지 |
| 넥스트 키 잠금 | 레코드 잠금 + 갭 잠금의 조합. 특정 행과 그 앞의 간격을 함께 잠금 |
| 범위 잠금 / 서술 잠금 | 조건에 해당하는 범위 전체에 거는 잠금 |

**갭 잠금 예시:**
```
row 1  ← 갭락
row 2  ← 갭락
row 3  ← 갭락
```
위아래로 갭 락이 걸려, 해당 범위 사이에 INSERT가 불가능해진다.

### InnoDB 레벨 락

| 잠금 종류 | 설명 |
|---|---|
| 공유 락 (S Lock, Shared Lock) | 읽기 락. 다른 트랜잭션도 읽기 가능, 쓰기는 불가 |
| 배타 락 (X Lock, Exclusive Lock) | 쓰기 락. 다른 트랜잭션 읽기/쓰기 모두 불가 |

> **결론**: MySQL에서 지원하는 락(레코드 락, 갭 락)과 InnoDB가 지원하는 락(S 락, X 락)은 동작 레이어가 다르다. 분류해서 이해해야 한다.
> - MySQL 5.6 이하 버전은 InnoDB가 없어 S 락, X 락을 지원하지 않는다.

## MVCC (Multi-Version Concurrency Control) 심화

- MVCC는 변경 전 원본 데이터를 **Undo Log** 공간에 스냅샷으로 저장한다.
- 읽기 시 스냅샷 데이터를 우선 참조하고, 스냅샷에 없는 데이터만 원본을 참조한다.
- **Repeatable Read 격리 레벨 이상에서만 MVCC가 동작**한다.

> MVCC = 원본은 놔두고, 스냅샷을 바라본다.

## 실습 코드 흐름 정리

### 1. Dirty Read 실습 (`READ_UNCOMMITTED`)

```
Thread A: updateStockAndForceRollback(productId, 10)
  └─ 재고를 10으로 수정 + saveAndFlush (커밋 X)
  └─ 5초 대기 (트랜잭션 열린 채로 유지)
  └─ setRollbackOnly() → 롤백

Thread B: getStockWithDirtyRead(productId)  [1초 후 실행]
  └─ READ_UNCOMMITTED로 조회
  └─ 커밋되지 않은 10을 읽음 → Dirty Read 발생!

최종 결과: Thread A가 롤백 → 재고는 원래 값(20)으로 복구
```

**핵심 포인트**: `saveAndFlush()`를 사용하는 이유
- 일반적으로 `@Transactional`은 메소드 종료 시 flush가 발생한다.
- 메소드가 끝나지 않은 상태(트랜잭션 열린 채)에서 DB에 쿼리를 보내려면 `saveAndFlush()`가 필요하다.

---

### 2. Non-Repeatable Read 실습 (`READ_COMMITTED`)

```
Thread A: demonstrateNonRepetableRead(productId)  [READ_COMMITTED 격리 수준]
  └─ 1차 조회: 재고 읽음 (예: 20)
  └─ 4초 대기
  └─ entityManager.clear() → 영속성 컨텍스트 비움
  └─ 2차 조회: 재고 다시 읽음

Thread B: updateStock(productId, 5)  [1초 후 실행]
  └─ 재고를 5로 변경 후 커밋

결과 비교: product1.getStock() != product2.getStock() → Non-Repeatable Read 발생!
```

**핵심 포인트**: `entityManager.clear()`를 사용하는 이유
- JPA 영속성 컨텍스트는 자체 스냅샷(1차 캐시)을 갖고 있어 같은 트랜잭션 내 재조회 시 DB가 아닌 캐시를 반환한다.
- `clear()`로 캐시를 비워야 실제 DB에서 다시 읽어와 격리 수준의 효과를 확인할 수 있다.

---

### 3. Phantom Read 실습 (`SERIALIZABLE` + 갭 락)

```
Thread A: demoSerializable()
  └─ SERIALIZABLE로 stock > 5인 상품 목록 조회
  └─ 갭 락 획득
  └─ 4초 대기
  └─ 다시 조회

Thread B: insertNewProduct("유령 신상품", 20)  [10초 후 실행]
  └─ 새 상품 INSERT 시도
  └─ 갭 락에 의해 블로킹 → 락 타임아웃 발생!
```

**결론**: `SERIALIZABLE`에서는 갭 락으로 인해 Thread B의 INSERT 자체가 차단되어 Phantom Read가 방지된다.

## 영속성 컨텍스트 vs MVCC

| | 영속성 컨텍스트 (JPA) | MVCC (DB) |
|---|---|---|
| 위치 | 애플리케이션(메모리) | 데이터베이스(Undo Log) |
| 스냅샷 기준 | 같은 트랜잭션 내 첫 조회 | 트랜잭션 시작 시점 |
| 동시성 제어 | X (캐시 역할만) | O |
| 격리 수준 적용 | X | O |

> 격리 레벨에 의한 동시성을 확인하려면 반드시 **DB 레벨**까지 내려가야 한다.
> JPA 영속성 컨텍스트가 스냅샷을 들고 있으면 DB 격리 수준의 효과가 가려진다.
> → `entityManager.clear()`로 캐시를 비워야 실제 동작을 확인할 수 있다.
