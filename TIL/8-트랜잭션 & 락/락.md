# 락 (Lock)

## 락(Lock) 종류

| 락 종류 | 설명 |
|---|---|
| 공유 락 (Shared Lock, S Lock) | 읽기 락. 다른 트랜잭션도 읽기 가능, 쓰기는 불가 |
| 배타 락 (Exclusive Lock, X Lock) | 쓰기 락. 다른 트랜잭션 읽기/쓰기 모두 불가 |
| 갭 락 (Gap Lock) | 범위에 새 행이 삽입되는 것을 방지 |

> 로우(Row) 수준의 접근은 락 시스템에서 관리하고,
> 격리 수준은 그 락을 어떻게 핸들링할지에 대한 정책이다.

## 왜 락이 필요한가?

Repeatable Read는 스냅샷 기반으로 읽기 일관성을 보장하지만,
동시에 같은 데이터를 읽고 쓰는 충돌은 막지 못한다.

```
Thread A: 재고 조회 → 10개
Thread B: 재고 조회 → 10개  (동시에 읽음)

Thread A: 10 - 1 = 9 저장
Thread B: 10 - 1 = 9 저장  (둘 다 10 기준으로 계산)

실제 결과: 9개  (원래는 8개여야 함) ← 데이터 정합성 깨짐
```

> 격리 수준은 "읽기 일관성" 문제를 해결하고,
> 락은 "동시 쓰기 충돌" 문제를 해결한다.

## 비관적 락 (Pessimistic Lock)

"어차피 충돌날거니까 처음부터 막자"

DB의 `SELECT ... FOR UPDATE`를 사용해 row에 X 락을 걸어
다른 트랜잭션이 해당 row에 접근하지 못하도록 차단한다.

### 동작 방식

```
Thread A: SELECT FOR UPDATE → row 잠금
Thread B: SELECT FOR UPDATE → 대기... (A가 끝날 때까지 블로킹)

Thread A: 재고 10 → 9 저장 후 커밋 → 락 해제
Thread B: 이제 읽음 → 재고 9 확인 → 8 저장

최종 결과: 8개 ← 정합성 보장
```

### JPA LockModeType 종류

| LockModeType | 발행 쿼리 | 설명 |
|---|---|---|
| `PESSIMISTIC_READ` | `SELECT ... FOR SHARE` | 공유 락. 다른 트랜잭션도 읽기 가능, 쓰기 불가 |
| `PESSIMISTIC_WRITE` | `SELECT ... FOR UPDATE` | 배타 락. 다른 트랜잭션 읽기/쓰기 모두 불가 |
| `PESSIMISTIC_FORCE_INCREMENT` | `SELECT ... FOR UPDATE` + version +1 | 배타 락 + 버전 강제 증가 (연관 엔티티 변경 감지용) |

일반적으로는 `PESSIMISTIC_WRITE`를 사용한다.  
`PESSIMISTIC_READ`는 읽는 동안 다른 트랜잭션의 쓰기만 막고 싶을 때 사용하지만, MySQL InnoDB에서는 `FOR SHARE`가 `FOR UPDATE`보다 데드락 위험이 있어 실무에서 잘 안 씀.

### JPA 구현

```java
// Repository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdWithLock(Long id);

// Service
@Transactional
public void decreaseStock(Long id) {
    Product product = productRepository.findByIdWithLock(id); // SELECT ... FOR UPDATE
    product.setStock(product.getStock() - 1);
}
```

### 장단점

| 장점 | 단점 |
|---|---|
| 확실하게 정합성 보장 | 대기가 생겨 성능 저하 |
| 구현이 단순 | 데드락(DeadLock) 위험 |

## 낙관적 락 (Optimistic Lock)

"충돌이 드물겠지, 충돌나면 그때 처리하자"

DB 락을 걸지 않고, 엔티티에 `@Version` 필드를 두어
수정할 때마다 버전을 +1 증가시킨다.
커밋 시점에 버전이 다르면 충돌로 판단하고 예외를 던진다.

### 동작 방식

```
Thread A: 재고 10 읽음 (version=1)
Thread B: 재고 10 읽음 (version=1)

Thread A: UPDATE ... WHERE version=1 → 성공! (version → 2)
Thread B: UPDATE ... WHERE version=1 → 실패! (이미 version=2)
          → OptimisticLockException 발생 → 재시도
```

### JPA 구현

```java
// Entity
@Entity
public class Product {

    @Version
    private Long version;  // 수정 시 자동으로 +1
}

// Service - 재시도 로직
@Retryable(retryFor = OptimisticLockingFailureException.class, maxAttempts = 3)
@Transactional
public void updateProduct(Long id) {
    Product product = productRepository.findById(id).orElseThrow();
    product.setStock(product.getStock() - 1);
}
```

### 예외 타입

낙관적 락 충돌 시 발생하는 예외가 두 가지로 나뉜다.

| 예외 | 발생 위치 | 설명 |
|---|---|---|
| `javax.persistence.OptimisticLockException` | JPA 표준 | JPA가 버전 충돌 감지 시 발생 |
| `org.springframework.dao.OptimisticLockingFailureException` | Spring | JPA 예외를 Spring이 감싼 것. 실제로 catch할 타입 |

`@Retryable`에서는 `OptimisticLockingFailureException`을 잡아야 한다.

### @Retryable 사용 시 주의사항

재시도할 때 **같은 트랜잭션 안에서 재시도하면 안 된다.**  
이미 예외가 발생한 트랜잭션은 롤백 마킹이 되어 있어 재사용할 수 없다.

```java
// 잘못된 예: @Transactional + @Retryable이 같은 메서드에 있으면
// 첫 번째 예외 후 트랜잭션이 rollback-only로 마킹되어 재시도 의미 없음
@Retryable(retryFor = OptimisticLockingFailureException.class, maxAttempts = 3)
@Transactional  // ← 위험
public void updateProduct(Long id) { ... }

// 올바른 예: 재시도 메서드와 트랜잭션 메서드를 분리
// Facade(재시도 담당) → Service(@Transactional)
@Retryable(retryFor = OptimisticLockingFailureException.class, maxAttempts = 3)
public void updateProductWithRetry(Long id) {
    productService.updateProduct(id); // @Transactional은 여기에
}
```

재시도마다 새 트랜잭션이 시작되어야 최신 version 값을 다시 읽어올 수 있다.

### 장단점

| 장점 | 단점 |
|---|---|
| 락을 안 걸어서 성능 좋음 | 충돌 시 예외 처리 + 재시도 로직 필요 |
| 데드락 없음 | 충돌이 잦으면 오히려 성능 저하 |

---

## 비관적 락 vs 낙관적 락

| | 비관적 락 | 낙관적 락 |
|---|---|---|
| 방식 | DB 락으로 선점 | 버전 번호로 충돌 감지 |
| 충돌 처리 | 사전 차단 (블로킹) | 사후 감지 (예외 + 재시도) |
| 성능 | 낮음 (대기 발생) | 높음 (락 없음) |
| 데드락 | 위험 있음 | 없음 |
| 적합한 상황 | 충돌이 빈번한 경우 | 충돌이 드문 경우 |

---

## 언제 뭘 쓸까?

| 상황 | 선택 |
|---|---|
| 재고 차감, 포인트 차감, 좌석 예약 | 비관적 락 (충돌 빈번, 정합성 최우선) |
| 게시글 수정, 설정 변경, 프로필 업데이트 | 낙관적 락 (충돌 드묾, 성능 중요) |
| 여러 서버(멀티 인스턴스)에서 동시 접근 | 분산 락 (DB 락은 단일 서버에만 유효) |

## 분산 락 (Distributed Lock)

비관적/낙관적 락은 **단일 DB 서버** 안에서만 유효하다.
서버가 여러 대로 스케일 아웃되면 각 서버가 서로의 락을 모르기 때문에 동시성 문제가 재발한다.

```
서버 A: DB에 비관적 락 획득 → 재고 10
서버 B: 서버 A의 락을 모름 → 동시에 재고 10 읽음 → 충돌!
```

이를 해결하기 위해 **외부 공유 저장소(Redis 등)** 에 락을 걸어 모든 서버가 같은 락을 바라보게 한다.

### MySQL Named Lock (GET_LOCK)

MySQL의 사용자 수준 락. 문자열 키로 락을 관리한다.  
별도 인프라 없이 DB만으로 분산 락처럼 쓸 수 있다.

```sql
SELECT GET_LOCK('product:1', 3);   -- 락 획득 (3초 대기)
-- ... 비즈니스 로직 ...
SELECT RELEASE_LOCK('product:1');  -- 락 해제
```

```java
// JPA Native Query로 구현
@Query(value = "SELECT GET_LOCK(:lockName, :timeout)", nativeQuery = true)
Integer getLock(String lockName, int timeout);

@Query(value = "SELECT RELEASE_LOCK(:lockName)", nativeQuery = true)
Integer releaseLock(String lockName);
```

- 장점: Redis 없이 DB만으로 분산 락 구현 가능
- 단점: DB 커넥션 자원을 락 용도로도 사용 → 커넥션 풀 고갈 위험

### Redis 분산 락

#### Lettuce - 스핀 락 방식

Lettuce는 기본 Redis 클라이언트(Spring Data Redis 기본 제공).  
`SET NX PX` 명령으로 락을 구현하며, 락을 못 얻으면 일정 시간마다 재시도(스핀 락)한다.

```
SET lock:product:1 "locked" NX PX 3000
# NX: 키가 없을 때만 SET (원자적 락 획득)
# PX 3000: 3초 후 자동 만료 (서버 죽어도 락 자동 해제)
```

```java
// Redis 스핀 락 예시
public void decreaseStock(Long id) throws InterruptedException {
    String lockKey = "lock:product:" + id;
    while (!redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "locked", 3, TimeUnit.SECONDS)) {
        Thread.sleep(100); // 100ms 후 재시도 (스핀)
    }
    try {
        // 비즈니스 로직
    } finally {
        redisTemplate.delete(lockKey); // 락 해제
    }
}
```

- 단점: 락을 못 얻으면 계속 재시도 → **Redis에 불필요한 부하** 발생

#### Redisson - pub/sub 방식

Redisson은 별도 Redis 클라이언트 라이브러리.  
스핀 락 대신 **pub/sub**을 사용해 락이 해제되면 구독자에게 알림을 보내 불필요한 재시도를 없앤다.

```java
// Redisson 사용
RLock lock = redissonClient.getLock("lock:product:" + id);

try {
    // 최대 5초 대기, 락 보유 3초
    if (lock.tryLock(5, 3, TimeUnit.SECONDS)) {
        try {
            // 비즈니스 로직
        } finally {
            lock.unlock();
        }
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

| | Lettuce | Redisson |
|---|---|---|
| 방식 | 스핀 락 (폴링) | pub/sub (이벤트) |
| Redis 부하 | 높음 (계속 재시도) | 낮음 (알림 대기) |
| 구현 복잡도 | 낮음 (기본 제공) | 높음 (별도 의존성) |
| 추천 | 락 경쟁이 적을 때 | 락 경쟁이 잦을 때 |

## 데드락 (DeadLock)

비관적 락 사용 시 주의해야 할 문제.

```
Thread A: Product(id=1) 잠금 → Product(id=2) 잠금 시도 (대기)
Thread B: Product(id=2) 잠금 → Product(id=1) 잠금 시도 (대기)

→ 서로가 서로를 기다려 영원히 진행 불가 = 데드락
```

### 방지 전략

**1. 락 획득 순서 고정**

여러 row에 락을 걸 때 항상 같은 순서(예: id 오름차순)로 획득하면 순환 대기가 생기지 않는다.

```java
// id가 작은 것부터 락 획득
List<Long> ids = Arrays.asList(id1, id2);
Collections.sort(ids); // 항상 오름차순으로 정렬 후 락 획득
```

**2. 락 타임아웃 설정**

일정 시간 내 락을 획득하지 못하면 포기하고 예외를 던져 데드락 상황을 탈출한다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@QueryHints(@QueryHint(name = "jakarta.persistence.lock.timeout", value = "3000")) // 3초 타임아웃
Optional<Product> findByIdWithLock(Long id);
```

**3. 트랜잭션을 짧게 유지**

락을 오래 잡을수록 데드락 확률이 높아진다.  
불필요한 외부 API 호출, 대용량 처리 등을 트랜잭션 밖으로 빼내 락 보유 시간을 최소화한다.

> MySQL InnoDB는 데드락을 자동 감지해 한 트랜잭션을 롤백시키지만, 이를 의존하기보다 설계 단계에서 방지하는 것이 좋다.
