# 연관관계 편의 메서드

양방향 관계에서 양쪽 객체 상태를 한 번에 동기화하는 메서드.

---

## 작성 규칙

| 항목 | 권장 |
|------|------|
| 위치 | FK 주인 쪽 (실제 DB 컬럼이 있는 쪽) |
| 이름 | `change~`, `add~` — 일반 setter와 구분, JPA 상태 변경임을 명시 |
| 역방향 동기화 | 반대편 객체도 항상 같이 세팅 |
| 기존 관계 정리 | `@ManyToOne`은 기존 컬렉션에서 `remove` 고려 |

---

## 코드 예시

**`@ManyToOne` — Member ↔ Team**
```java
// Member.java (FK 주인)
public void changeTeam(Team team) {
    if (this.team != null) {
        this.team.getMembers().remove(this); // 기존 팀에서 제거
    }
    this.team = team;
    team.getMembers().add(this); // 반대편 동기화
}
```

**`@OneToOne` — Order ↔ Delivery**
```java
// Order.java (FK 주인)
public void addDelivery(Delivery delivery) {
    this.delivery = delivery;
    delivery.setOrder(this); // 반대편 동기화
}
```

> `@OneToMany` 관계는 Many 쪽(FK 주인)에 넣는 게 원칙이지만,
> 실무에서는 더 자주 호출되는 쪽에 두기도 한다.

---

## 양방향은 언제 설정할까?

### 기본 원칙
**단방향으로 먼저 설계하고, 필요해질 때 양방향으로 열어준다.**

연관관계 편의 메서드는 양방향 관계일 때만 의미 있다.
단방향은 반대편을 참조하지 않으므로 동기화할 대상 자체가 없기 때문.

---

### 양방향이 필요한 경우

1. **역방향 조회가 빈번할 때**
```java
// "이 배송의 주문 정보가 뭐야?" 를 자주 조회해야 할 때
delivery.getOrder() // 이런 탐색이 자주 필요하면 양방향 고려
```

2. **화면/비즈니스 로직 양쪽에서 탐색이 필수일 때**
```java
// 주문서 화면 → order.getDelivery()
// 배송 관리 화면 → delivery.getOrder()
// 둘 다 자주 쓰인다면 양방향 설정
```

---

### 단방향으로 충분한 경우 (대부분)

역방향 조회가 필요해도 JPQL로 해결 가능하면 단방향으로 충분하다.

```java
// Order → Delivery 단방향만 있어도
// "이 배송의 주문?" 은 JPQL로 조회
em.createQuery("select o from Order o where o.delivery = :delivery")
```

---

### 양방향의 단점 (함부로 쓰면 안 되는 이유)

| 문제 | 설명 |
|------|------|
| 연관관계 편의 메서드 필수 | 빠뜨리면 1차 캐시 불일치 버그 발생 |
| 무한루프 위험 | `toString()`, JSON 직렬화 시 서로 참조해서 StackOverflow |
| 관리 포인트 증가 | 양쪽 모두 신경 써야 해서 복잡도 상승 |

---

### 요약

```
단방향 설계 → 역방향 탐색이 실제로 필요해질 때 → 그때 양방향으로 추가
```

처음부터 양방향으로 설계하지 말 것.
