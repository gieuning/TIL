# Docker 이미지 태그

## 기본 사용법

```bash
docker build -t 이미지이름:태그 .

docker build -t my-app:latest .
docker build -t my-app:1.0.0 .
```

태그를 생략하면 자동으로 `latest`가 붙는다.

---

## latest의 문제

`latest`는 "가장 최근 빌드"라는 관례적 이름일 뿐, Docker가 자동으로 최신을 추적하지 않는다.

```
빌드 1: my-app:latest  → v1 코드
빌드 2: my-app:latest  → v2 코드  (v1은 추적 불가)
```

운영 환경에서 `latest`만 사용하면:
- 이전 버전으로 롤백이 어렵다.
- 어느 코드가 배포됐는지 태그만 보고 알 수 없다.

---

## 좋은 태그 전략

| 방식 | 예시 | 특징 |
|---|---|---|
| 날짜 + 시간 | `release-260428-1500` | 사람이 읽기 쉬움 |
| Git commit hash | `app-3f2a91c` | 코드와 1:1 추적 가능 |
| CI/CD 빌드 번호 | `build-1024` | 파이프라인과 연결 |

좋은 태그의 조건:
- 순서를 알 수 있다.
- 고유하다.
- 사람이 매번 수동으로 관리하지 않아도 된다.

## Semantic Versioning

```
MAJOR.MINOR.PATCH
예: 1.4.2
```

| 올리는 경우 | 버전 |
|---|---|
| 이전 버전과 호환되지 않는 변경 | MAJOR |
| 이전 버전과 호환되는 기능 추가 | MINOR |
| 버그 수정 | PATCH |

### 서버 배포 이미지에서 잘 안 쓰는 이유

- MAJOR/MINOR/PATCH 기준이 서버 변경에는 애매하다.
- 매 배포마다 사람이 수동으로 버전을 올려야 한다.
- 실제 변경 내용은 Git 로그/PR로 확인하면 되므로 태그에 의미를 실을 필요가 적다.

> 라이브러리/SDK처럼 외부에 API를 제공하는 경우엔 Semantic Versioning이 중요하다.  
> 서버 배포 이미지는 날짜, Git hash, 빌드 번호가 더 실용적이다.
