# Docker 내부 통신 정리

## 핵심 개념: 내부 통신 vs 외부 통신

컨테이너 간 통신 방식에 따라 **보안 수준**과 **오버헤드**가 달라진다.

### 외부 통신 (비효율)
```
svc3 → ping → 네트워크 밖으로 나감 → 다시 네트워크 안으로 → 192.0.0.59(svc4)
```
- IP로 접근하면 같은 네트워크 레이어에서 직접 접근 불가
- 불필요하게 망 밖을 나갔다 들어오는 오버헤드 발생
- IP 접근은 외부에서도 가능하다는 의미 → 보안에 취약

### 내부 통신 (효율적)
```
svc3 → ping → svc4 (같은 네트워크 내 이름으로 직접 접근)
```
- 같은 네트워크 안에 있으면 이름(도메인)으로 바로 접근 가능
- 오버헤드 없음, 망 밖으로 나가지 않음

## Docker 네트워크 종류별 특징

### Bridge (기본값)
- `docker run` 시 네트워크 지정 없으면 자동으로 bridge
- **IP 통신은 가능**, **이름(DNS) 통신은 불가**
- 컨테이너 이름으로 서로 찾지 못함

```bash
docker run -d --name svc1 --network bridge busybox sleep infinity
docker run -d --name svc2 --network bridge busybox sleep infinity

docker exec -it svc1 ping svc2
# 결과: ping: bad address 'svc2'  ← DNS 해석 실패
```

### 사용자 정의 네트워크 (권장)
- **이름(도메인)으로 통신 가능** → DNS 해석 지원
- 같은 네트워크에 묶인 컨테이너끼리 이름으로 직접 통신

```bash
docker network create my-network

docker run -d --name svc3 --network my-network busybox sleep infinity
docker run -d --name svc4 --network my-network busybox sleep infinity

docker exec -it svc3 ping svc4
# 성공 ← 같은 네트워크이므로 DNS 해석됨
```

### Host 네트워크
- 호스트(내 PC) 네트워크와 컨테이너 네트워크가 동일하게 동작
- 외부 IP로 바로 접근 가능
- 개발 환경에서 편리하게 사용

```
외부IP:80 → 컨테이너:80 == 내 PC 네트워크
```

## 내부 통신을 써야 하는 이유

### 1. 보안
- 외부에서 접근 못하도록 포트/IP 전부 막아두고
- 하나의 네트워크 안에 서비스를 묶어 내부에서만 접근 가능하게 설정

### 2. SSL 인증서 불필요
```
# MSA 내부 통신
API 서버 → http://내부-도메인 → 다른 API 서버
```
- 내부끼리 통신하면 SSL 인증서가 필요 없음 (`http`로만 통신)
- 외부로 나가는 구간에만 SSL 적용

### 3. 오버헤드 감소
- OSI 7계층을 타지 않고 내부에서 직접 라우팅
- 불필요한 네트워크 왕복이 없어 속도 향상

## 실제 아키텍처 예시

### SSR 프론트엔드 + 백엔드

**비효율적인 방식 (외부 경유)**
```
클라이언트 → 프론트 서버(SSR) → https://api.sparta.com → API 서버
                                  ↑ 외부 망으로 나갔다가 돌아옴
```

**효율적인 방식 (내부 통신)**
```
클라이언트 → 프론트 서버(SSR) → http://내부-도메인 → API 서버
                                  ↑ 내부망 안에서 직접 통신
```

### K8s / MSA 전체 구조
```
외부 요청 흐름:
클라이언트 → DNS → nginx (SSL 인증서 적용) → 로드밸런서 → 내부 API 서버

내부 통신 흐름:
내부 API 서버 → http://내부-도메인 → 다른 내부 API 서버 (SSL 불필요)
```

### Next.js BFF 패턴
```
클라이언트 → nginx → Next.js 서버 (내부망) → API 서버들 (내부 통신)
```
- RESTful API 설계로 한 페이지에서 여러 API 동시 호출이 많아지는 문제
- Next.js가 중계 역할(BFF)을 하며 내부에서 여러 API를 묶어 클라이언트에 단일 응답 제공
- 외부 접근은 모두 nginx를 통해서만 허용

## 핵심 정리

| 구분 | Bridge (기본) | 사용자 정의 네트워크 |
|------|--------------|------------------|
| IP 통신 | 가능 | 가능 |
| 이름(DNS) 통신 | 불가 | **가능** |
| 보안 | 낮음 | 높음 |
| 권장 환경 | 테스트 | 운영/개발 |

> **이름(도메인)으로 통신해야 진정한 내부 통신이다.**  
> IP 접근은 외부에서도 가능하다는 의미이므로, 같은 네트워크에서 이름으로 통신하도록 설정하는 것이 보안과 효율 모두에서 유리하다.
